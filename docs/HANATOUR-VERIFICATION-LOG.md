# 하나투어 실측 검증 로그 (HANATOUR Verification Log)

## 1. 문서 목적

- **설계안을 늘리는 문서가 아니다.** 브라우저·F12·실제 응답을 기준으로 확인한 사실을 **날짜별로 누적**한다.
- 다음 축에 대해 **확정 / 미확인 / 반려**를 구분해 기록한다.
  - `originCode` / `supplierGroupId` / `supplierDepartureCode` 후보
  - 출발일 수집 방식(달력·팝업·API·HTML 등)
  - 일정 탭·원문 구조
  - 차단·헤더·로그인 필요 여부
- **Product / ProductDeparture / ItineraryDay** 역할 분리를 전제로 하며, 한 덩어리로 “추측 매핑”하지 않는다.
- VERYGOODTOUR(fragment)·MODETOUR(API) 패턴에 **억지로 끼워 맞춘 서술을 하지 않는다.**

---

## 2. 사용 방법

1. 실측할 때마다 **§4 템플릿을 복사**해 새 블록을 **문서 하단(또는 최신 로그 위)** 에 붙인다.
2. **추정**은 “추정”이라고 적고, **Network 응답·URL·스크린샷 번호** 등 재현 근거를 짧게 남긴다.
3. 같은 주제를 다시 보면 **기존 항목을 덮어쓰지 말고** 새 로그 행을 추가하고, §5 요약 표만 갱신한다.
4. 템플릿의 **출발일 구조 유형 A~E**는 본 문서 **§2** 표의 정의를 따른다.

### 출발일 구조 유형 정의 (A~E)

| 코드 | 의미 (실측 시 해당하는 것만 체크) |
|------|-----------------------------------|
| **A** | 인페이지 달력(월 전환 시 DOM 위주) |
| **B** | 모달·레이어·팝업 형 출발일 선택 UI |
| **C** | 별도 페이지·iframe 등 URL/컨텍스트 전환 |
| **D** | XHR/fetch 등 **네트워크 JSON**이 주 데이터 소스 |
| **E** | 서버 렌더 HTML에 출발일 표가 한 번에 포함 |

복합이면 `A+D`처럼 **복수 표기** 가능.

---

## 3. 상태 표기 규칙

| 표기 | 의미 |
|------|------|
| **확인됨** | 동일 조건에서 **2회 이상** 재현했거나, 응답 본문·URL로 **근거가 명확**함. |
| **미확인** | 아직 보지 않았거나, 1회만 봤고 재현·근거가 부족함. |
| **반려됨** | 실측 결과 **해당 후보/가설이 틀림**이 명확함 (근거 한 줄 필수). |
| **보류** | 사이트 변경·차단·시간 부족 등으로 **당장 판단 보류** (다음 액션 명시). |

**원칙:** selector 문자열·API 경로를 **“확인됨”으로 적으려면** 재현 절차와 함께 적는다. 근거 없이 적지 않는다.

## 4. 실측 로그 템플릿 (복사용)

아래 블록을 복사해 사용한다.

```markdown
### 로그 — YYYY-MM-DD

| 항목 | 내용 |
|------|------|
| **날짜** | YYYY-MM-DD |
| **확인자** | (이름 또는 이니셜) |
| **대상 URL** | (전체 URL) |
| **페이지 유형** | 상세 / 출발일 팝업 / 일정 탭 / 기타: ___ |
| **originCode 후보** | 값: ___ / 상태: 확인됨·미확인·반려됨·보류 / 근거: ___ |
| **supplierGroupId 후보** | 값: ___ / 상태: ___ / 근거: ___ |
| **supplierDepartureCode 후보** | 값: ___ / 상태: ___ / 근거: ___ |
| **출발일 구조 유형 (A~E)** | (예: B+D) |
| **월 이동 방식** | (예: 화살표 클릭 / URL 쿼리 변경 / API 파라미터만 변경 등) — 상태: ___ |
| **row 유일키 후보** | (날짜만으로 구분되는지, 추가 ID 필요 여부) — 상태: ___ |
| **일정 원문 확보 가능 여부** | 확인됨·미확인·반려됨 / ItineraryDay 단위로 나뉘는지: ___ |
| **차단·로그인·헤더** | 없음 / 로그인 필요 / 특정 Referer·쿠키 필요 / 기타: ___ — 상태: ___ |
| **첨부 메모** | (스크린샷 경로, HAR 파일명, 요청 이름 패턴 등 — selector 단정 문구 지양) |
| **최종 판정** | (이번 세션에서 이 URL·유형에 대해 어떤 결론까지 갔는지 한 줄) |
```

## 5. 누적 판정 요약 표

**규칙:** 로그가 쌓일 때마다 이 표만 최신 합의로 수정한다. 셀 안에도 **확인됨 / 미확인 / 반려됨 / 보류**를 쓸 수 있다.

| 축 | 누적 판정 (최신 기준) | 비고 |
|----|----------------------|------|
| **originCode SSOT 후보** | 미확인 | 화면 표기 상품번호 ≠ DB `originCode` 로 쓸 값이라고 가정하지 않음 |
| **supplierGroupId 후보** | 미확인 | |
| **supplierDepartureCode 후보** | 미확인 | |
| **출발일 수집 구조** | 미확인 | A~E 또는 복합 |
| **일정 수집 구조** | 미확인 | 탭·lazy·원문 단위 |
| **구현 가능성 / 남은 리스크** | 미확인 | 차단·정책 변경·모바일 분기 등 |

---

## 6. 금지 사항

- **추정한 selector**를 “확인됨”으로 기록하거나, 그 내용을 **코드에 그대로 반영하지 않는다.**
- **화면에 보이는 상품번호**를 근거 없이 **`originCode` SSOT로 확정하지 않는다.**
- **일정 요약 문장만**으로 **ItineraryDay 행을 임의 생성**했다고 기록하거나 구현 가정으로 삼지 않는다. (원문·일차 구조 실측 필수)
- Product에 **출발일별 가격·좌석**을 넣는 식의 역할 붕괴 서술을 하지 않는다.
