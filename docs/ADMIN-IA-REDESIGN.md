# 봉투어 관리자 시스템 — IA/UX 재설계안

실제 코드 구조(`AdminSidebar`, `/admin`, `/admin/control`, `/admin/dashboard`, `/admin/pending`, `/admin/products`, `/admin/scheduler-settings`, `/admin/bookings`)를 반영한 운영 플로우 기준 재정의안.

---

## 1. 전체 IA(정보구조) 최종안

### 현재 구조 (사이드바 기준)

| 순서 | 메뉴명     | 실제 경로              | 비고                          |
|------|------------|------------------------|-------------------------------|
| 1    | 대시보드   | /admin/control         | 오늘 수집·봇 상태·로그만 있음 |
| 2    | 홈         | /admin                 | 저장·시세·상담·등록상품 등 집합 |
| 3    | 등록대기   | /admin/pending         | pending → registered 전환     |
| 4    | 상품 목록  | /admin/products        | 운영 상품 목록                |
| 5    | 상품 등록  | /admin/dashboard       | URL/텍스트·Gemini·저장        |
| 6    | 스케줄러·보안 | /admin/scheduler-settings | Cron·비상정지 등           |

- **회원관리** 메뉴 없음. 예약/상담은 홈에서 “예약 관리 보기”로 `/admin/bookings` 링크만 존재.

---

### 추천 개편안 (최종)

**메뉴 6개 + 회원관리 포함을 위해 “홈”을 운영 허브로 통합하고, 대시보드와 역할을 나눔.**

| 순서 | 메뉴명       | 경로 제안        | 한 줄 정의 |
|------|--------------|------------------|------------|
| 1    | **대시보드**  | /admin 또는 /admin/control 통합 | 오늘 한눈에: 수집 현황·등록대기 N건·상품목록·상담 접수·빠른 액션 |
| 2    | **상품 등록**  | /admin/register | URL/텍스트 붙여넣기 → Gemini 분석 → 저장 → 등록대기로 이동 |
| 3    | **등록대기**  | /admin/pending   | 이미지 수급·검수·승인 후 상품목록으로 승격 |
| 4    | **상품 목록**  | /admin/products  | 승인 완료 상품·가격 동기화·노출/비노출·고객 메인 노출 관리 |
| 5    | **회원·예약**  | /admin/members (신규) 또는 /admin/bookings 확장 | 회원·상담 신청·예약 요청·관리자 계정·권한 |
| 6    | **스케줄러·보안** | /admin/scheduler-settings | Cron·즉시 1회 실행·비상정지·보안 설정 |

**왜 이렇게 바꾸는가**

- **대시보드 vs 홈**: “대시보드”는 KPI·빠른 액션만 두고, “홈”에 있던 저장/시세/상담/등록상품은 각각 **상품 등록·등록대기·상품 목록·회원·예약**으로 이전. 메뉴에서 “홈”을 제거하고 **대시보드 = 관리자 진입 첫 화면(운영 허브)** 으로 하나로 통합.
- **회원관리 축**: “회원·예약” 메뉴를 신규 추가. 예약(bookings)은 그 하위 또는 동일 페이지 내 탭으로 두고, 추후 회원(User)·권한 확장 가능하도록 이름을 “회원·예약”으로 정의.
- **상품 등록**: “상품 등록” 하나로 명확히. URL/텍스트 입력·저장 후 **등록대기로 이동** 안내.

---

### 최종 추천안 요약

- **안 A (홈 유지)** 는 하지 않음. 홈과 대시보드 역할이 겹치므로, **안 B: 홈 제거 + 대시보드 = 운영 허브 + 회원·예약 추가** 를 **최종 추천안**으로 확정.

---

## 2. 메뉴별 역할 정의

### 대시보드 (통합 후)

- **메뉴명**: 대시보드
- **목적**: 관리자 로그인 후 첫 화면. 오늘 할 일·현황·빠른 액션만 제공.
- **꼭 있어야 하는 것**: 등록대기 N건, 상품목록 N건, 오늘 수집 성공/실패, 상담 접수 N건, “상품 등록하기” “등록대기 보기” “가격 동기화 1회 실행” 등 빠른 버튼.
- **있으면 안 되는 것**: 긴 폼(텍스트 붙여넣기·상품 저장 폼), 시세 상세 테이블(2주치), 상담 포인트 상세, 등록 상품 전체 테이블. → 이들은 각각 상품 등록·등록대기·상품 목록·회원·예약 페이지로.
- **상위/하위**: 최상위. 하위 페이지 없음.
- **주요 행동**: 숫자 확인 → “등록대기 보기” “상품 등록” “예약 보기” 클릭.
- **다른 메뉴와 구분**: “보기만 하고 빠르게 이동”하는 허브. 실제 입력·검수·승인은 모두 다른 메뉴에서.

---

### 홈 (제거)

- **메뉴명**: (삭제)
- **목적**: 기존 “홈” 기능은 대시보드(요약) + 상품 등록 + 등록대기 + 상품 목록 + 회원·예약으로 재배치.
- **꼭 있어야 하는 것**: 없음(페이지 제거).
- **있으면 안 되는 것**: 새 기능 추가하지 않음.
- **상위/하위**: 없음. `/admin` 리다이렉트를 `/admin`(대시보드)로 통일.

---

### 상품 등록

- **메뉴명**: 상품 등록
- **목적**: 여행사 선택 → URL 또는 텍스트/HTML 붙여넣기 → AI 분석 → 미리보기 → 저장 → 등록대기로 이동.
- **꼭 있어야 하는 것**: 여행사(브랜드) 선택, URL 입력 또는 텍스트/HTML 붙여넣기, “분석” 버튼, 미리보기(상품명·가격·일정 요약), “저장” 버튼, 저장 후 “등록대기로 이동” 안내 및 링크.
- **있으면 안 되는 것**: 시세 관제, 상담 포인트, 등록 상품 목록, 예약 현황, 스케줄러 설정.
- **상위/하위**: 대시보드에서 진입. 저장 후 이동 대상 = 등록대기.
- **주요 행동**: 붙여넣기 → 분석 → 확인 → 저장.
- **다른 메뉴와 구분**: “새 상품을 한 건 만드는” 단일 책임. 검수·이미지·승인은 등록대기/상품목록.

---

### 등록대기

- **메뉴명**: 등록대기
- **목적**: 저장된 상품의 이미지 수급·검수·승인 후 상품목록(고객 노출 가능)으로 올리는 단계.
- **꼭 있어야 하는 것**: pending 상태 상품 리스트(여행사·코드·상품명·이미지 준비 여부·등), 행별 “이미지 수급” “검수” “승인(등록)” 버튼, 승인 시 registrationStatus → registered 전환.
- **있으면 안 되는 것**: 상품 새로 저장하는 폼, 가격 동기화 대량 실행, 시세 관제.
- **상위/하위**: 상품 등록 다음 단계. 승인 후 이동 = 상품 목록.
- **주요 행동**: 이미지 채우기 → 검수 → 승인.
- **다른 메뉴와 구분**: “이미 저장된 상품 중 아직 승인 전”만. 승인된 상품은 상품 목록에서만.

---

### 상품 목록

- **메뉴명**: 상품 목록
- **목적**: 승인 완료(registered) 상품 관리. 고객 메인 노출 대상. 수정·가격 동기화·노출/비노출.
- **꼭 있어야 하는 것**: 상품 테이블(여행사·코드·상품명·목적지·노출 여부 등), 검색/필터, 행별 “상세 수정” “가격 동기화” “노출/비노출” 전환, “등록대기로 되돌리기”는 선택.
- **있으면 안 되는 것**: pending 전용 “승인” 플로우(그건 등록대기).
- **상위/하위**: 등록대기 승인 후 도착지. 상세 수정 = /admin/products/[id].
- **주요 행동**: 검색·필터 → 수정·동기화·노출 전환.
- **다른 메뉴와 구분**: “이미 고객에게 보여줄 수 있는” 상품만. pending은 등록대기에서만.

---

### 회원·예약 (신규)

- **메뉴명**: 회원·예약
- **목적**: 상담 신청·예약 요청·회원(추후)·관리자 계정·권한을 한 축에서 관리.
- **꼭 있어야 하는 것**: 예약(상담) 목록, 상태(접수완료/상담중/예약확정/취소), 상세 보기·상태 변경. 추후: 회원 목록·관리자 계정·역할(ADMIN 등).
- **있으면 안 되는 것**: 상품 등록·등록대기·상품 목록·스케줄러 설정.
- **상위/하위**: 대시보드에서 “상담 N건” 클릭 시 여기로. 상세 = /admin/bookings/[id] 또는 모달.
- **주요 행동**: 예약/상담 목록 확인 → 상태 변경.
- **다른 메뉴와 구분**: “사람(회원·예약자·관리자)” 중심. 상품·스케줄은 다른 메뉴.

---

### 스케줄러·보안

- **메뉴명**: 스케줄러·보안
- **목적**: 가격 수집 봇 실행 주기·즉시 1회 실행·비상정지·보안 관련 설정.
- **꼭 있어야 하는 것**: Cron 시간 설정, min/max delay, UA 랜덤·마우스 이동·headless 등 옵션, “지금 1회 실행”, “비상정지”, (선택) 최근 실행 로그/상태.
- **있으면 안 되는 것**: 상품 CRUD, 회원/예약 관리, 시세 상세 테이블.
- **상위/하위**: 독립. 실행 결과는 대시보드 “오늘 수집 현황” 또는 전용 로그 영역에서 확인.
- **주요 행동**: 설정 변경 저장, run once, 비상정지.
- **다른 메뉴와 구분**: “자동화·보안 설정”만. 상품/회원 데이터는 건드리지 않음.

## 3. 최종 메뉴 구조 추천 (2안 비교)

### 안 A: 홈 유지안

- **구성**: 대시보드, **홈**, 등록대기, 상품 목록, 상품 등록, 스케줄러·보안, 회원·예약(7개).
- **장점**: 기존 “홈” 사용 습관 유지.
- **단점**: 홈과 대시보드 역할이 겹치고, “홈”에 기능이 계속 몰려 구조가 불명확.
- **운영 적합성**: 낮음. 중복과 책임 분산.
- **최종 추천**: 아니오.

### 안 B: 홈 제거 + 회원·예약 추가 (최종 추천안)

- **구성**: **대시보드**(운영 허브), **상품 등록**, **등록대기**, **상품 목록**, **회원·예약**, **스케줄러·보안** (6개).
- **장점**: 플로우가 “등록 → 대기 → 목록 → 노출”로 명확하고, 회원·예약 축이 생기며, 대시보드는 “한눈에 보기 + 빠른 이동”만 담당.
- **단점**: 기존 “홈” 진입 경로를 바꿔야 함. 기존 홈 URL(/admin)을 대시보드로 쓰면 이전과 동일 진입점 유지 가능.
- **운영 적합성**: 높음.
- **최종 추천**: **예. 최종 추천안.**

---

## 4. 최종 추천안 기준 상세 페이지 설계

### [대시보드]

- **페이지 목적**: 관리자 첫 화면. 오늘 할 일·핵심 숫자·빠른 액션만 제공.
- **상단 헤더 문구**: "봉투어 관리자"
- **서브 문구**: "오늘 현황과 빠른 작업"
- **핵심 KPI/카드** (예시):
  - 등록대기 **N**건 → 클릭 시 등록대기
  - 상품 목록 **N**건 (노출 중 **M**건)
  - 오늘 수집 **성공 K / 실패 L**
  - 상담 접수 **P**건 → 클릭 시 회원·예약(또는 예약 목록)
- **주요 섹션**: KPI 카드 1행, “빠른 액션” 버튼 그룹(상품 등록, 등록대기 보기, 가격 동기화 1회, 예약 보기), (선택) 최근 수집 로그 요약.
- **주요 버튼**: [상품 등록하기], [등록대기 보기], [상품 목록 보기], [가격 동기화 1회 실행], [예약·상담 보기].
- **리스트/테이블**: 대시보드 본문에는 긴 테이블 없음. 카드와 버튼만.
- **상태값/배지**: 없음(요약 숫자만).
- **액션 흐름**: 숫자 확인 → 필요한 메뉴로 이동.
- **다른 페이지로 이동**: 위 버튼으로 /admin/register, /admin/pending, /admin/products, /admin/members(또는 /admin/bookings), (가격 동기화는 API 호출 또는 스케줄러 run once).
- **빈 상태**: "등록대기 0건, 상품 0건. 상품 등록에서 첫 상품을 추가해 보세요."
- **에러 상태**: "데이터를 불러오지 못했습니다. 새로고침해 주세요."
- **성공 상태**: (저장 등은 이 페이지에서 하지 않으므로 별도 토스트는 하위 페이지에서.)

---

### [상품 등록]

- **페이지 목적**: URL 또는 텍스트로 상품 한 건을 분석·저장하고 등록대기로 보냄.
- **상단 헤더 문구**: "상품 등록"
- **서브 문구**: "URL 또는 상세 텍스트를 붙여넣어 AI가 분석한 뒤 저장합니다. 저장 후 등록대기에서 이미지·검수를 진행하세요."
- **핵심 KPI/카드**: 없음.
- **주요 섹션**:
  1. 여행사(브랜드) 선택 (드롭다운)
  2. 입력: URL 1개 또는 “텍스트/HTML 붙여넣기” 텍스트 영역
  3. [분석하기] 버튼 → Gemini 호출
  4. 미리보기: 상품명, 목적지, 가격 요약, 일정 요약
  5. [DB에 저장] 버튼
  6. 저장 후 문구: "저장되었습니다. 등록대기에서 이미지 수급 후 승인해 주세요." + [등록대기로 이동] 링크
- **주요 버튼**: [분석하기], [DB에 저장], [등록대기로 이동].
- **리스트/테이블**: 없음(단일 상품 입력).
- **상태값/배지**: 분석 중, 저장 중 (로딩).
- **액션 흐름**: 여행사 선택 → URL 또는 텍스트 입력 → 분석 → 미리보기 확인 → 저장 → 등록대기 링크 클릭.
- **다른 페이지로 이동**: 저장 후 등록대기. 상단에 [대시보드], [등록대기], [상품 목록] 링크.
- **빈 상태**: "URL을 입력하거나 상세 텍스트를 붙여넣은 뒤 분석하기를 눌러 주세요."
- **에러 상태**: "분석 실패: {메시지}" / "저장 실패: {메시지}"
- **성공 상태**: "저장되었습니다. 등록대기에서 이미지 수급 후 승인해 주세요."

### [등록대기]

- **페이지 목적**: pending 상품의 이미지·검수·승인. 승인 시 상품 목록으로.
- **상단 헤더 문구**: "등록대기"
- **서브 문구**: "저장된 상품에 이미지를 채우고 검수한 뒤 승인하면 상품 목록(고객 노출)으로 이동합니다."
- **핵심 KPI/카드**: 등록대기 **N**건 (상단 요약).
- **주요 섹션**: 테이블(여행사, 코드, 상품명, 이미지 준비, 업데이트일), 행별 액션: [이미지 수급], [상세 보기], [승인(등록)].
- **주요 버튼**: [이미지 수급](→ 상품 상세 또는 이미지 워크플로우), [승인(등록)].
- **리스트/테이블 컬럼**: 여행사, 상품코드, 상품명, 목적지, 이미지 준비(Y/N 또는 N/N), 업데이트일, 액션.
- **상태값/배지**: 이미지 준비됨/미준비, (선택) 검수 완료.
- **액션 흐름**: 목록 확인 → 이미지 수급(필요 시) → 승인 → 해당 행 목록에서 제거(또는 “상품 목록에서 보기” 안내).
- **다른 페이지로 이동**: 승인 후 상품 목록. [상품 등록], [상품 목록], [대시보드].
- **빈 상태**: "등록대기 상품이 없습니다. 상품 등록에서 새 상품을 저장해 주세요."
- **에러 상태**: "목록을 불러오지 못했습니다."
- **성공 상태**: "승인되었습니다. 상품 목록에서 확인하세요."

## 5. 핵심 운영 플로우 설계

### A. 상품 등록 플로우

1. 상품 등록 페이지 진입.
2. 여행사(브랜드) 선택.
3. URL 입력 또는 텍스트/HTML 붙여넣기.
4. [분석하기] → Gemini 호출 → 미리보기(상품명·가격·일정 요약).
5. [DB에 저장] → API 저장.
6. 성공 시: "저장되었습니다. 등록대기에서 이미지 수급 후 승인해 주세요." + [등록대기로 이동] 버튼.
7. 사용자가 [등록대기로 이동] 클릭 → 등록대기 페이지.

### B. 등록대기 플로우

1. 등록대기 목록에서 상품 선택.
2. 이미지 필요 시: [이미지 수급] → 상품 상세(/admin/products/[id]) 또는 전용 이미지 UI에서 Pexels/생성/업로드.
3. 검수(내부 기준) 후 [승인(등록)] 클릭.
4. API: registrationStatus → 'registered'. 해당 행은 등록대기 목록에서 제거(또는 “상품 목록에서 보기” 안내).
5. 상품 목록에서 해당 상품 확인.

### C. 상품 목록 플로우

1. 상품 목록에서 검색/필터(항공사·목적지·에러만).
2. 수정: [상세 수정] → /admin/products/[id].
3. 가격 동기화: 행별 또는 선택 일괄 [가격 동기화] → API 호출.
4. 노출: [노출/비노출] 토글 → 고객 메인 노출 여부 반영.
5. (선택) 등록대기로 되돌리기: registrationStatus → 'pending'.

### D. 회원·예약 플로우

1. 회원·예약(또는 예약) 목록에서 접수 건 확인.
2. 상세 보기 → 고객명·연락처·상품·인원·금액·상태.
3. 상태 변경: 접수완료 → 상담중 → 예약확정 또는 취소.
4. (추후) 회원 목록·관리자 계정·ADMIN 권한 부여.

### E. 스케줄러 플로우

1. 매일 실행 시간(Cron 시·분) 설정.
2. restMin/restMax, UA·마우스·headless 설정 후 [설정 저장].
3. [지금 1회 실행] → API 호출 → 결과는 대시보드 또는 로그에 표시.
4. [비상 정지] → 실행 중 프로세스 종료 신호.

## 6. 상태 체계(state machine) 제안

| 상태 | 의미 | 보이는 페이지 | 다음 액션 | 누가 변경 |
|------|------|----------------|-----------|-----------|
| pending | 저장됨, 등록 전(이미지·검수 대기) | 등록대기 | 이미지 수급, 검수, 승인(등록) | 관리자 |
| registered | 승인 완료, 상품 목록에 노출 가능 | 상품 목록 | 가격 동기화, 노출/비노출, 상세 수정 | 관리자 |
| (선택) hidden | 상품 목록에 있으나 고객 메인 비노출 | 상품 목록 | 노출 전환 | 관리자 |
| (선택) error | 수집/파싱 오류 | 상품 목록(필터) | 재시도, 수정 | 자동/관리자 |

- **DRAFT/PARSED**: 현재 코드에서 명시적으로 쓰이지 않으면 제거하거나, “저장 직후 pending”으로 통일.
- **APPROVED**: “registered”와 동일 의미로 사용(이미 registrationStatus = 'registered').
- **PUBLISHED**: “고객 메인 노출 여부”는 별도 필드(예: isPublished 또는 노출 플래그)로 두고, 상품 목록에서 토글로 변경.

## 8. 현재 홈 화면 요소 재배치안

| 현재 요소 | 추천 이동 대상 | 이유 | 유지/이동/삭제 |
|-----------|----------------|------|----------------|
| 저장(DB에 저장) | 상품 등록 | 새 상품 저장은 상품 등록 전용 페이지에서 | **이동** |
| 여행사 선택 | 상품 등록 | 동일 | **이동** |
| 텍스트 붙여넣기 | 상품 등록 | 동일 | **이동** |
| DB에 저장 버튼 | 상품 등록 | 동일 | **이동** |
| 최근 1시간 상품 이미지 생성 | 등록대기 또는 상품 등록 하단 | 이미지 수급은 등록대기와 연관 | **이동** (등록대기 또는 상품 등록) |
| 브랜드(여행사) 관리 | 상품 등록 내 링크 또는 대시보드 빠른 링크 | 브랜드는 상품 등록 시 선택과 직결 | **유지** (상품 등록 + 대시보드 링크) |
| 상품 등록(상세) 페이지로 이동 | 제거(이름만 정리) | “상품 등록”이 메뉴로 있으므로 “상세 등록”은 /admin/register 등으로 통일 | **이동** (링크 문구만 “상품 등록”으로) |
| 가격 동기화 지금 실행 | 대시보드 빠른 액션 또는 스케줄러 | 1회 실행은 대시보드/스케줄러에서 | **이동** |
| 시세 관제 | 상품 목록 또는 상품 상세 | 시세는 특정 상품 기준이므로 목록/상세에서 | **이동** |
| 상담 포인트 | 상품 상세(/admin/products/[id]) | 상품별 상담 포인트 | **이동** |
| 등록 상품(여행사/코드) | 상품 목록 | 전체 상품 목록은 상품 목록 페이지에서 | **이동** |
| 실시간 상담 현황 | 회원·예약(예약 목록) | 상담 접수 = 예약/상담 관리 | **이동** |

## 9. 실제 구현 우선순위

1. **대시보드 정리**  
   - /admin(또는 /admin/control)을 “대시보드” 하나로 통합.  
   - KPI 카드(등록대기 N, 상품 N, 오늘 수집, 상담 N) + 빠른 액션 버튼만 두고, 나머지 섹션 제거 또는 링크로 대체.

2. **사이드바 메뉴 변경**  
   - “홈” 제거. “대시보드” 클릭 시 /admin(대시보드)으로.  
   - “회원·예약” 추가 → /admin/bookings 또는 /admin/members.  
   - “상품 등록” 경로 통일: /admin/dashboard 또는 /admin/register 중 하나로, 라벨은 “상품 등록”.

3. **상품 등록 페이지 정리**  
   - 현재 홈에 있는 저장 폼(AdminProductPasteForm 등)을 상품 등록 페이지로 이동.  
   - 저장 후 “등록대기로 이동” 안내 및 링크 추가.

4. **등록대기 이미지 워크플로우**  
   - “최근 1시간 이미지 생성” 등을 등록대기 또는 상품 등록 하단에 배치.  
   - 등록대기 테이블에 “이미지 준비” 컬럼·행별 [이미지 수급] 액션 정리.

5. **상품 목록 공개 흐름**  
   - 노출/비노출 필드·API·UI 반영.  
   - 시세 관제는 상품 목록 또는 상품 상세로 이동.

6. **회원·예약 페이지**  
   - /admin/bookings를 사이드바 “회원·예약”으로 노출.
