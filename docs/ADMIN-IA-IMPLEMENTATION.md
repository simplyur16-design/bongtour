# 봉투어 관리자 시스템 — IA/UX 개편 실행안

**전제**: B2C 상담유도형 사이트. 직접예약/결제 없음. 고객은 상품 조회 후 상담예약·문의 남김. 초기 채널은 카카오 오픈채팅 중심. 관리자 IA는 이에 맞춰 “상담 접수·상태 관리”에 초점을 둠.

**기준 문서**: docs/ADMIN-IA-REDESIGN.md  
**실제 경로**: /admin, /admin/control, /admin/pending, /admin/products, /admin/dashboard, /admin/scheduler-settings, /admin/bookings

---

## 1. 최종 추천 IA 확정안

### 최종 메뉴 6개

| 순서 | 최종 명칭 | 실제 사용 경로 | 한 줄 정의 |
|------|-----------|----------------|------------|
| 1 | **대시보드** | `/admin` | 오늘 현황(KPI)과 빠른 액션만. 관리자 첫 화면. |
| 2 | **상품 등록** | `/admin/register` | URL/텍스트 붙여넣기 → AI 분석 → 저장 → 등록대기로 이동. |
| 3 | **등록대기** | `/admin/pending` | 이미지 수급·검수·승인 후 상품 목록으로 승격. |
| 4 | **상품 목록** | `/admin/products` | 승인 완료 상품. 고객 메인 노출·가격 동기화·노출 관리. |
| 5 | **상담·예약** | `/admin/bookings` | 상담 접수·상태 관리. (직접결제 없음, 상담유도형) |
| 6 | **스케줄러·보안** | `/admin/scheduler-settings` | 가격 수집 봇 Cron·즉시 1회·비상정지·보안 옵션. |

### 확정 이유

- **대시보드 = /admin**: 첫 진입을 하나로 통일. `/admin/control` 내용(오늘 수집·봇 상태·로그)을 `/admin`에 흡수해 “대시보드” 하나만 유지.
- **상품 등록 = /admin/register**: `/admin/register` 페이지는 이미 존재(parse-and-register 플로우). 사이드바만 “상품 등록”을 `/admin/register`로 연결하면 됨. `/admin/dashboard`(대형 폼)는 “상세 등록” 대안으로 두거나 `/admin/register`로 리다이렉트.
- **회원·예약 → 상담·예약**: 직접예약/결제가 없고 “상담 접수 → 카카오 등으로 연락” 구조이므로, 메뉴명을 **상담·예약**으로 고정. “회원”은 추후 로그인 회원이 생길 때 확장. 경로는 기존 `/admin/bookings` 유지(추가 라우트 불필요).
- **홈 제거**: “홈” 메뉴 삭제. `/admin`이 곧 대시보드.

## 2. 홈/대시보드 정리안

### /admin 과 /admin/control 처리

| 항목 | 결정 | 설명 |
|------|------|------|
| **최종 대시보드 경로** | `/admin` | 관리자 진입 시 항상 `/admin`으로. |
| **/admin 페이지** | **유지·역할 변경** | 기존 “홈” 내용(저장·시세·상담·등록상품) 제거 후, **대시보드 전용**으로 교체. KPI 카드 + 빠른 액션만 표시. 필요 시 `/admin/control`의 “오늘 수집·봇 상태·로그” 컴포넌트를 여기로 가져옴. |
| **/admin/control** | **흡수 후 제거 또는 리다이렉트** | 대시보드 내용을 `/admin`에 넣은 뒤, `/admin/control`은 `/admin`으로 301/302 리다이렉트. 또는 control의 컴포넌트만 `/admin/page.tsx`로 이전하고 control 라우트 삭제. |

**권장**: `/admin/page.tsx`를 대시보드용으로 전면 수정. `/admin/control/page.tsx`의 ScraperStatusIndicator·오늘 수집 현황·AdminLogTerminal·run once 버튼 등을 `/admin`으로 옮긴 뒤, `/admin/control`은 `redirect('/admin')` 처리.

### 현재 홈(/admin) 요소 재배치 표

| 현재 요소 | 옮길 페이지 | 유지/이동/삭제 | 이유 |
|-----------|-------------|----------------|------|
| 저장(DB에 저장) | 상품 등록 `/admin/register` | **이동** | 새 상품 저장은 상품 등록 단일 페이지 책임. |
| 여행사 선택 | 상품 등록 | **이동** | 동일. |
| 텍스트 붙여넣기 | 상품 등록 | **이동** | 동일. |
| DB에 저장 버튼 | 상품 등록 | **이동** | 동일. |
| 최근 1시간 상품 이미지 생성 | 등록대기 `/admin/pending` 또는 상품 등록 하단 | **이동** | 이미지 수급은 등록대기와 연관. |
| 브랜드(여행사) 관리 | 상품 등록 내 링크 + 대시보드 빠른 링크 | **유지(위치만 변경)** | 상품 등록 시 선택과 직결. 대시보드에는 “브랜드 관리” 링크만. |
| 상품 등록(상세) 페이지로 이동 | — | **삭제(문구만)** | “상품 등록” 메뉴로 통일. 링크는 [상품 등록] 버튼으로 대체. |
| 가격 동기화 지금 실행 | 대시보드 빠른 액션 | **이동** | 대시보드에서 1회 실행 버튼만. |
| 시세 관제 | 상품 목록 또는 상품 상세 `/admin/products/[id]` | **이동** | 상품 단위 시세이므로 목록/상세에서. |
| 상담 포인트 | 상품 상세 `/admin/products/[id]` | **이동** | 상품별 데이터. |
| 등록 상품(여행사/코드) 테이블 | 상품 목록 `/admin/products` | **이동** | 전체 상품은 상품 목록에서. |
| 실시간 상담 현황 + 예약 관리 보기 | 상담·예약 `/admin/bookings` | **이동** | 대시보드에는 “상담 접수 N건” 카드 + [상담·예약 보기] 버튼만. |

## 3. 회원·예약 메뉴 재정의 (상담·예약)

### /admin/bookings 확장/흡수

- **경로 유지**: `/admin/bookings`. 새 경로 `/admin/members`는 만들지 않음.
- **역할**: “상담 접수 건” 목록·상세·상태 변경. 직접예약/결제 UI는 넣지 않음.
- **확장**: 사이드바에 **상담·예약** 메뉴 추가하고, 기존 bookings 페이지를 이 메뉴의 유일한 화면으로 사용. 추후 “고객 목록”(문의 이력만) 탭 추가 가능.

### 메뉴명 비교 (B2C 상담유도형 기준)

| 후보 | 적합성 | 비고 |
|------|--------|------|
| 회원·예약 | 보통 | “회원”이 강조되면 로그인·가입이 있는 것처럼 보일 수 있음. 현재는 상담 접수 중심. |
| 상담·예약 | **가장 적합** | 상담 접수·상담중·예약확정(담당자 확정) 흐름과 일치. 결제 엔진 아님을 드러냄. |
| 고객·상담 | 적합 | “고객”만 강조할 때 좋음. “예약”이 빠지면 “예약확정” 상태가 메뉴명에 안 드러남. |

**최종 추천 메뉴명**: **상담·예약**

### 이 페이지에서 다룰 데이터 (현재 단계)

| 데이터 | 포함 여부 | 비고 |
|--------|------------|------|
| 상담 접수 | ✅ | Booking 목록. 고객명·연락처·상품·인원·희망일·접수일. |
| 예약 요청 | ✅ | 동일 Booking. “예약확정”은 “담당자 확정·카카오 등으로 연락” 의미. 결제 아님. |
| 고객 기본정보 | ✅ | 접수 건에 포함된 이름·전화·이메일. 별도 “회원” 테이블 없어도 됨. |
| 관리자 계정/권한 | ❌ (추후) | 현재 단계에서는 Auth/관리자 로그인만. “회원관리” 탭은 미구현. |

**직접예약/결제 없는 범위**: 결제 금액·결제 수단·PG 연동·예약금/잔금 UI는 넣지 않음. “상담 접수 → 상태 변경(접수완료/상담중/예약확정/취소)”만 관리.

## 4. 페이지별 상세 설계

### 대시보드 (`/admin`)

| 항목 | 내용 |
|------|------|
| **페이지 목적** | 관리자 첫 화면. 오늘 숫자 확인 후 필요한 메뉴로 빠르게 이동. |
| **헤더 문구** | "봉투어 관리자" |

| **주요 섹션** | KPI 카드 1행, 빠른 액션 버튼 그룹, (선택) 오늘 수집 로그 요약·봇 상태. |
| **주요 버튼** | [상품 등록], [등록대기 보기], [상품 목록 보기], [가격 동기화 1회], [상담·예약 보기], [브랜드 관리](링크). |
| **표 컬럼** | 없음(카드·버튼만). |
| **상태값/배지** | 없음. |
| **빈 상태 문구** | "등록대기 0건, 상품 0건. 상품 등록에서 첫 상품을 추가해 보세요." |
| **성공 문구** | (이 페이지에서 저장 없음. 가격 동기화 1회 시 "요청을 보냈습니다." 등 토스트만.) |
| **에러 문구** | "데이터를 불러오지 못했습니다. 새로고침해 주세요." |
| **다른 페이지로 이동** | 상품 등록(/admin/register), 등록대기(/admin/pending), 상품 목록(/admin/products), 상담·예약(/admin/bookings), 브랜드(/admin/brands). |
| **절대 들어가면 안 되는 기능** | 텍스트 붙여넣기·상품 저장 폼, 시세 2주 테이블, 상담 포인트 상세, 등록 상품 전체 테이블, 예약 상세 편집 폼. |

| **절대 들어가면 안 되는 기능** | 시세 관제, 상담 포인트, 상품 목록 테이블, 예약 현황, 스케줄러 설정. |

## 5. 실제 운영 플로우 재확정 (B2C 상담유도형)

- **상품 등록**: 여행사 선택 → URL 또는 텍스트 붙여넣기 → Gemini 분석 → 저장 → **등록대기로 이동** 안내. 관리자는 “한 건씩 저장”만.
- **등록대기**: 이미지 수급(Pexels/생성/업로드) → 검수 → **승인** → 상품 목록으로 이동. 관리자는 “이미지 채우기 + 승인”만.
- **상품 목록**: 승인된 상품만 표시. **노출/비노출**·가격 동기화·상세 수정. 고객 메인 노출은 “노출”인 상품만. 관리자는 결제/재고가 아닌 **노출 여부**만 관리.
- **상담 접수**: 고객이 상담예약/문의 제출 → Booking 생성 → 관리자는 **접수완료/상담중/예약확정/취소** 상태만 변경. 결제·입금 확인 UI 없음. “예약확정”은 “담당자가 상담 확정·카카오 등으로 연락” 의미.
- **고객 메인페이지 노출**: 상품 목록에서 “노출”인 상품만 메인에 표시. 관리자는 상품 목록에서 토글로 제어.
- **스케줄러 실행**: Cron에 따라 가격 수집 봇 실행. 관리자는 스케줄러·보안에서 시간·옵션·run once·비상정지만.

**관리자가 관리하는 상태만 정리**: (1) 상품: pending ↔ registered, 노출 on/off. (2) 상담: 접수완료·상담중·예약확정·취소. 직접예약/결제·재고·정산은 현재 단계에서 다루지 않음.

## 6. 상태 체계 제안

### 상품 상태

| 상태 | 의미 | 보이는 페이지 | 다음 가능한 액션 | 자동/수동 |
|------|------|----------------|------------------|-----------|
| pending | 저장됨, 등록 전(이미지·검수 대기) | 등록대기 | 이미지 수급, 검수, 승인(등록) | 수동(관리자) |
| registered | 승인 완료, 상품 목록에 있음 | 상품 목록 | 가격 동기화, 노출/비노출, 상세 수정 | 수동(관리자) |
| (노출 플래그) | 고객 메인 노출 여부 | 상품 목록 | 노출 on/off 토글 | 수동(관리자) |
| (선택) error | 수집/파싱 오류 | 상품 목록 필터 | 재시도, 수정 | 자동/수동 |

- **DRAFT/PARSED/PENDING_IMAGE/PENDING_REVIEW**: 현재 코드에 없으면 도입하지 않음. **pending** 하나로 “등록대기” 통일.
- **APPROVED**: **registered**와 동일. registrationStatus = 'registered'.
- **PUBLISHED**: “고객 메인 노출”은 별도 필드(예: isPublished 또는 노출 플래그)로 두고, **PUBLISHED**라는 상태값보다는 “노출 on”으로 표현.
- **HIDDEN**: “노출 off”로 표현. 상태값을 늘리지 않고 플래그로 처리 가능.

### 상담(Booking) 상태

| 상태 | 의미 | 다음 가능한 액션 | 자동/수동 |
|------|------|------------------|-----------|
| 접수완료 | 접수됨, 미처리 | → 상담중, 취소 | 수동 |
| 상담중 | 담당자 상담 진행 중 | → 예약확정, 취소 | 수동 |
| 예약확정 | 상담 확정(카카오 등 연락 완료). **결제 아님** | → 취소(필요 시) | 수동 |
| 취소 | 취소됨 | (종료) | 수동 |
