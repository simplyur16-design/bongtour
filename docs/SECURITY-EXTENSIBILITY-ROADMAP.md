# 보안/확장 실행 로드맵 (관리자 중심 단계)

## 전제
- 현재 단계의 중심은 관리자 페이지/운영 흐름이다.
- 고객 페이지 UI/UX는 아직 미확정이다.
- 따라서 실행 항목은 시점별로 분리해 진행한다.

---

## 1) 지금 당장 해야 하는 것 (관리자 중심 구조 고정 단계)

### A. 공개 API 응답 스키마 점검(1차)
- 대상: `/api/gallery`, `/api/featured`, `/products/[id]` 직렬화 경로.
- 점검 포인트:
  - `imageManualSelected`, `imageSelectionMode`, `imageCandidateOrigin`, 내부 notes, usage log가 응답에 포함되지 않는지 재검증.
  - 공개 응답 DTO를 문서/타입으로 명시.

### B. 관리자 API 보호 기본선 강화
- `/api/admin/*` rate limit 1차 도입(고정 윈도우 또는 토큰 버킷).
- 우선 민감 엔드포인트부터 적용:
  - 이미지 생성/변환/업로드
  - 스케줄 재처리
  - 상품 승인/상태 변경

### C. 감사 로그 조회 권한 분리 설계
- `AssetUsageLog`는 존재하므로 조회 권한 계층을 정의:
  - `ADMIN` 기본 조회 가능
  - 향후 `SUPER_ADMIN` 전용 상세(PII/원문 notes 포함) 분리

### D. misuse/abuse 탐지 1차
- 비정상 호출 패턴 감지 기준을 최소 정의:
  - 단시간 다량 요청
  - 실패율 급증
  - 관리자 엔드포인트 연속 401/403
- 임시로 서버 로그 기반 알림 기준부터 문서화.

---

## 2) 고객 페이지 UI/UX 확정 직전에 해야 하는 것

### A. 공개 응답 계약 확정(2차)
- 고객 상세/목록/갤러리용 DTO를 확정하고, 관리자 메타 완전 차단 재확인.
- 공개 API 스냅샷 테스트 추가(메타 유출 회귀 방지).

### B. 예약/상담 진입 파라미터 규칙 확정
- 상품 단위 진입 vs 상품+출발일 진입 규칙 확정.
- 아직 UI 구현은 하지 않고 파라미터 계약만 고정.

### C. 카카오 상담 추적 범위 결정
- 단순 링크인지, 상품/출발일/문의유형 추적을 포함할지 결정.

---

## 3) 예약하기 구현 직전에 해야 하는 것

예약은 자동 예약이 아니라 **고객정보 접수 -> 운영자 수동 예약 진행** 구조를 유지한다.

### A. 입력 스키마/검증 확정
- 필수 입력:
  - 이름
  - 전화번호
  - 이메일
  - 총 인원
  - 성인/아동/유아 구분
  - 아동/유아 생년월일
  - 선택/희망 출발일
  - 상품 식별값(`productId`, `originSource`, `originCode`)
  - 요청사항
- 가능하면 `ProductDeparture` 식별값 연계.

### B. PII 정책 확정
- 수집 최소화 원칙
- 저장 시 마스킹 규칙(목록/상세 분리)
- 보관기간/파기 정책(자동 파기 배치 포함)
- 접근 권한(관리자 역할별 조회 범위) 확정

### C. 알림 체계 확정
- 관리자 알림: 카카오 + 이메일
- 알림 필드:
  - 고객명/연락처/이메일
  - 상품명/공급사
  - 희망(선택) 출발일
  - 인원 구성
  - 아동/유아 생년월일
  - 요청사항
  - 관리자 바로가기 링크
- 고객 확인 메시지:
  - “예약 요청 접수 확인”만 발송
  - 확정/결제/혜택 확정 문구 금지

---

## 4) 혜택 기능 구현 직전에 해야 하는 것

### A. 우선순위 정책 확정 (Brand vs Product vs 기간성)
- 권장 우선순위:
  1. 기간성 프로모션(유효기간 내)
  2. Product 단위 혜택
  3. Brand 단위 상시 혜택
- 충돌 시 우선순위와 병합 규칙(중복 문구/중복 카드) 확정.

### B. 혜택 데이터 검증/게시 정책
- 게시 상태(검수 완료만 노출)
- 시작/종료일 유효성
- 공급사/상품 매핑 무결성

### C. 예약/상담과의 연결 방식 확정
- 예약 접수 시 “당시 노출 혜택 스냅샷” 저장 여부 결정(분쟁 대응 목적).

---

## 5) 공개 오픈 직전에 해야 하는 것

### A. 보안 하드닝 최종 점검
- 공개 API 응답 메타 유출 스캔(자동 + 수동)
- `/api/admin/*` rate limit 운영값 튜닝
- 관리자 세션/권한 우회 시나리오 점검

### B. abuse 탐지/알림 운영화
- 임계치 기반 알림(슬랙/이메일/카카오 중 택1 이상)
- 장애 대응 런북(누가, 언제, 어떻게 차단/해제하는지)

### C. 개인정보/감사 운영 체크
- PII 보관/파기 실제 동작 확인
- 접근 로그/감사 로그 조회 권한 최종 확인

---

## 예약하기 기능 착수 시점 권고
- 아래 3조건을 모두 만족할 때 1차 구현 착수:
  1. 고객 상세 UI 흐름(진입/입력/완료)이 대체로 확정
  2. `Product` / `ProductDeparture` 운영 안정도 확보
  3. PII 정책 + 관리자 알림 템플릿 확정

권고: 현재 단계에서는 UI를 미루고, 서버 입력 스키마/알림 계약부터 선구현하는 방식이 안전하다.
