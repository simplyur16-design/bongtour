# 관리자 중심 구조 고정 (보안/확장)

## 현재 범위 고정
- 현재 단계의 핵심은 관리자 운영 흐름이다: 등록대기, 승인, day 이미지 선택, 라이브러리, 자산 사용 로그, 공급사 수집/검수.
- 고객 페이지(UI/UX)는 미확정이다. 따라서 현재는 관리자/데이터/보안 구조를 우선 고정한다.
- Product / ProductDeparture / ItineraryDay 3축 구조는 유지한다.
- Prisma 런타임/생성 정책은 `docs/PRISMA-RUNTIME-POLICY.md`를 단일 기준으로 따른다.

## 보안 원칙 (현재 코드 기준)

### 1) 공개/관리자 경로 분리
- 관리자 경로는 `/admin/*` 및 `/api/admin/*`로 분리한다.
- 미들웨어 보호 범위는 관리자 라우트만 대상으로 유지한다.
- 공개 페이지(`/products/*`)에는 관리자 우회 인증 쿠키/파라미터 로직을 적용하지 않는다.

### 2) 내부 메타 비공개
- 공개 렌더 경로에서 `schedule` 내부 메타(`imageManualSelected`, `imageSelectionMode`, `imageCandidateOrigin`)는 기본 비노출한다.
- 내부 메타는 관리자 API/화면에서만 사용한다.

### 3) 서버 중심 로직 유지
- 공급사 adapter, 이미지 후보 선정, 승인/검수, usage log 기록은 서버에서 수행한다.
- 프론트에는 결과 데이터만 전달한다.

### 4) 원본/서비스 자산 분리
- 업로드 시 서비스용 WEBP 정규화 자산을 생성하여 사용한다.
- 공개 페이지에는 서비스용 경로만 사용하고, 내부 로그/판단 메타는 노출하지 않는다.

### 5) 관리자 API 보호
- `/api/admin/*`는 `requireAdmin()` 강제.
- 향후 보강 우선순위: rate limit, 감사 로그 조회 권한 세분화, abuse 탐지.

## 향후 고객 페이지 확장 포인트 (UI 미구현)

> 예약 접수 정책은 `docs/BOOKING-INTAKE-POLICY.md`, 예약 상태·전이는 `docs/BOOKING-STATUS-POLICY.md`를 단일 기준으로 사용한다.
> 시기별 실행 순서는 `docs/SECURITY-EXTENSIBILITY-ROADMAP.md`를 기준으로 진행한다.

### 예약하기 (고객 접수 -> 운영자 수동 예약)
- 데이터 연결 기준:
  - Product (상품)
  - ProductDeparture (희망 출발일/회차)
  - Booking 또는 별도 BookingRequest (접수 전용)
- 최소 접수 필드 후보:
  - 이름, 연락처, 이메일, 인원, 희망 출발일, 문의사항
- 관리자 처리 흐름:
  - 접수 목록 확인 -> 수동 예약 진행 -> 상태 변경

### 1:1 카카오 상담
- 2가지 모드 지원 가능하도록 서버 파라미터 설계:
  - 공통 링크
  - 상품/출발일/문의유형 포함 링크
- 추적 필요 시 상담 진입 로그를 별도 테이블로 분리 가능.

### 여행사별 혜택 / 상품별 혜택
- Brand 단위 혜택 + Product 단위 혜택으로 분리 설계가 자연스럽다.
- 기간성 혜택은 별도 promotion 엔터티(시작/종료/우선순위)로 확장 가능.
- 공개 노출 전에는 관리자 검수 상태(게시 가능 여부) 필드 필요.

## 선행 체크리스트 (고객 UI 구현 전)
- 공개 API 응답 스키마에서 관리자 전용 메타 제외 확인
- 예약/상담 입력 API의 PII 처리 정책(마스킹/보관기간) 정의
- 관리자 API rate limit 및 abuse 대응 기본 정책 반영
- 혜택 데이터 소스(Brand/Product/기간성) 우선순위 규칙 확정
