# 예약(Booking) 상태값·전이 정책

코드 기준: `lib/booking-status-policy.ts` · 관리자 PATCH `app/api/admin/bookings/[id]/route.ts`

## 전제

- 예약은 **자동 확정이 아님**. 고객 정보 접수 후 운영자가 수동으로 진행한다.
- 상태 변경은 **인증된 관리자**만 API로 수행한다(`requireAdmin`).

## 최종 상태 목록

| 상태 | 의미 |
|------|------|
| `접수완료` | `/api/bookings` 저장 직후 기본값 |
| `상담중` | 담당자 연락·문의 대응 중 |
| `예약진행중` | 공급사/좌석 등 예약 처리 중 |
| `예약확정` | 고객에게 확정 안내 가능한 단계(정책상 최종 확정) |
| `보류` | 일정·인원·서류 등 보류(재개 전까지 처리 보류) |
| `취소` | 요청 취소·무응답 종료 등 |

## 허용 전이 (from → to)

- **접수완료** → 상담중, 예약진행중, 보류, 취소  
- **상담중** → 예약진행중, 예약확정, 보류, 취소  
- **예약진행중** → 예약확정, 보류, 취소  
- **예약확정** → 취소  
- **보류** → 접수완료, 상담중, 예약진행중, 취소  
- **취소** → 접수완료, 상담중 (**재개** — 운영상 동일 요청 재개 시)

동일 상태로의 PATCH는 **성공(노이즈 방지)**로 처리한다.

## 관리자 처리 흐름(권장)

1. 접수 알림(카카오·이메일) 확인 → `접수완료`  
2. 고객 연락 시작 → `상담중`  
3. 공급사 예약/좌석 확보 진행 → `예약진행중`  
4. 조건 합의 → `예약확정`  
5. 일정 변경·서류 대기 등 → `보류` 후 위 단계로 복귀  
6. 고객·공급사 사유로 취소 → `취소` (재협의 시 `취소` → `접수완료` 또는 `상담중`)

## 예외·운영 메모 (문서/매뉴얼)

| 상황 | 권장 |
|------|------|
| 동일 고객 중복 요청 | 별도 `Booking` row로 관리. 필요 시 메모에 기존 건 번호 기재. |
| 잘못된 출발일/매진 | 고객 안내 후 새 요청 유도 또는 `보류`에서 일정 확정 후 진행. |
| 취소 후 재개 | 상태만 `취소` → `접수완료` 또는 `상담중`으로 올리고, 금액·일정은 재확인. |
| 출발일 변경 | 향후 스키마에 `selectedDate` 수정 API가 붙기 전까지는 운영 메모·별도 처리. |

## UI

- 관리자 예약 화면: `getNextBookingStatuses(현재상태)` 기준으로 다음 가능한 상태만 버튼 표시.

## DB 점검·정규화 (레거시 문자열)

- 조회: `npm run booking:statuses:inspect` (`scripts/inspect-booking-statuses.ts`) — `BOOKING_STATUSES` 외 값이 있으면 종료 코드 `2`
- trim만 필요: `npm run booking:statuses:normalize:dry-run` → `...:apply`
- 알 수 없는 값 일괄 매핑(운영자 확인 후):  
  `npx tsx scripts/normalize-booking-statuses.ts --apply --map-unknown-to 접수완료`
